## 1. `useFileSelection` — 610 рядків без жодного тесту

### Що було не так

`hooks/use-file-selection.ts` — найбільший рукописний файл у репозиторії й єдине
місце, де живе весь конвеєр між `ContentSource` і готовим PDF. Обидва режими,
анонімний і GitHub, ганяють саме цей хук.

Чисті шматки, які він складає, тести мали: `paginate`, `selectFolder`,
`resolveStatuses`, `estimatePages`. Не мала їх **оркестрація** — те, як воно
складається докупи. І обидва баги, які знайшлися руками в застосунку, жили саме
там, а не в чистих функціях.

### Чому зелений тест на готовому коді нічого не вартий

Коли пишеш тести до вже написаного коду, вони проходять одразу. Це нічого не
доводить: тест міг би пройти й на зламаному коді.

Тому кожне з трьох ключових місць я ламав назад і дивився, чи набір це помітить:

| Що зламано                                           | Скільки тестів впало |
| ---------------------------------------------------- | -------------------- |
| Прибрано позначку `unsupported` при збої декодування | 4                    |
| Папка рахується сумою замість одного потоку          | 2                    |
| Вимкнено кеш рендеру за сигнатурою                   | 1                    |

Три з трьох. Після кожної мутації файл відновлювався; `git diff` на хуку був
порожній.

### Дві розбіжності, які знайшлися по дорозі

**Про `.DS_Store` я помилявся в кращий бік.** Написав тест виходячи з того, що
після збою папка лишається наполовину виділеною і другий клік її добирає. Тест
впав: хук доходить до повного виділення **з першого кліку**. Бінарник одночасно
вилітає з виділення і перестає рахуватися серед доступних, тож чекбокс папки
відразу стає повним. Переписав тест під фактичну поведінку і додав другий: цикл
«зняти → вибрати» не перечитує файл повторно.

**Один тест був моєю помилкою у фікстурі.** Поклав один файл у `src/`, і
`commonRoot` з'їв префікс — override записався як `a.ts`, а не `src/a.ts`. Це
правильна поведінка: оверрайди зберігаються відносно кореня репозиторію.
Виправив фікстуру, не код.

### Що тепер покрито

Найцінніше — **NDA-обмеження тепер перевіряється, а не мається на увазі**.
`describeSelection` має повертати рівно `path`, `contentHash`, `sizeBytes`, і
тест звіряє список ключів. Байт вмісту туди не просочиться непоміченим.

Далі: пошук репо-конфігу відносно спільного кореня (це баг із префіксом при
drag-and-drop), точний рахунок папки не показується, поки не виміряні **всі**
її файли, попередження перед додаванням `used`/`vendored` без блокування,
зняття файлу ніколи не попереджає, прев'ю закривається при зміні виділення.

### Компроміс, який варто пам'ятати

Мок рівно один — `usePdfWorker`, бо він конструює справжній `Worker` з URL
модуля, а це межа, яку тест перетнути не може. Метрики шрифту підставні. Але
`paginate`, `estimatePages`, `selectFolder`, `resolveStatuses`,
`decodeSourceFile` і `sha256Hex` працюють по-справжньому, а всі очікувані числа
сторінок виводяться викликом того самого `paginate` з тими самими метриками.
Тобто тести перевіряють хук, а не геометрію сторінки.

`jsdom` вмикається докблоком `// @vitest-environment jsdom` у самому файлі, а не
в конфізі — так набори з pdfkit лишаються в `node`.

## 2. Приватний репозиторій віддавав 429 «GitHub throttling»

### Симптом

`GET /api/github/tree?owner=GopkoDev&repo=AI-stotel` → `429 Too Many Requests`,
повідомлення «GitHub is throttling rapid requests. Wait a moment and retry.»
Публічний репозиторій відкривався нормально.

### Причина

`lib/github/errors.ts` — одна зайва умова:

```ts
if (retryAfter || response.status === 429 || remaining !== null) {
  return { kind: "secondary-rate-limit" }
}
return { kind: "forbidden" } // ← недосяжно
```

GitHub чіпляє заголовки `x-ratelimit-*` до **кожної** відповіді, включно з
помилками. Тож `remaining !== null` істинне завжди, і гілка `forbidden` була
мертвим кодом. Приватний репозиторій без доступу віддає
`403 Resource not accessible by integration`, а воно потрапляло в «throttling»,
і `statusForError` перетворював це на 429.

Порада «зачекай і повтори» в цьому випадку не могла спрацювати ніколи: жодне
очікування не дає доступу до репозиторію.

### Чому тести це пропустили

Тест `maps a plain 403 to forbidden` існував і був зелений. Його фікстура —
`response(403)` **без жодного заголовка**. GitHub так ніколи не відповідає.
Тест перевіряв ситуацію, якої не буває.

Це загальний урок: фікстура, спрощена до нереалістичності, дає хибну впевненість
краще, ніж відсутність тесту взагалі.

### Фікс

Тролінг тепер читається лише з явної 429, заголовка `retry-after`, або
вичерпаного бюджету (`remaining: 0`). Наявність заголовків більше нічого не
означає.

Записано в `.claude/ARCHITECTURE.md` у розділ «Notes that are easy to get wrong».

## 3. Чому доступу все одно не було — дозволи GitHub App

Після фікса повідомлення стало правдивим (403), але репозиторій усе ще не
відкривався. Діагноз склався з чотирьох спостережень:

| Виклик                                 | Потрібен дозвіл                          | Результат        |
| -------------------------------------- | ---------------------------------------- | ---------------- |
| `/user/installations`                  | жодного                                  | ✅ вхід працює   |
| `/user/installations/:id/repositories` | `metadata` (видається автоматично)       | ✅ список репо є |
| `git/trees` **публічного** репо        | жодного — публічне читає будь-який токен | ✅ працює        |
| `git/trees` **приватного** репо        | **`contents: read`**                     | ❌ 403           |

Єдина конфігурація, за якої сходяться всі чотири рядки одночасно — у App не
виставлено `Contents: Read-only`. `docs/SPEC.md:191` цього вимагає, а
`docs/tasks/todo.md:253` (реєстрація App) досі стоїть невиконаною.

**Головна пастка тут:** зміна дозволів не діє на вже встановлені інсталяції
автоматично. GitHub чекає підтвердження на
`github.com/settings/installations` — без цього кроку симптом не зміниться.

Успіх на публічному репозиторії нічого не доводить про дозволи, бо публічні дані
читає будь-який автентифікований токен. Це збиває з пантелику найбільше.

### Побічний фікс

Повідомлення про 403 тепер називає **обидві** можливі причини — і відсутній
дозвіл, і невиданий репозиторій. На рівні 403 їх не розрізнити без зайвого
запиту до API, а порада лише про другу відправляла б перетикати репозиторій,
який і так відмічений.

## 4. Base UI: `Button` як посилання втрачав семантику кнопки

### Симптом

У консолі на кожен рендер:

> Base UI: A component that acts as a button expected a native `<button>`
> because the `nativeButton` prop is true.

З `AppLayout` (чотири рази), `RepoWorkspace`, `RepoList`.

### Причина

Base UI має проп `nativeButton` зі значенням `true` за замовчуванням — тобто
«`render` віддає справжній `<button>`». Проєкт рендерить `Button` як `Link` або
`<a>` у дванадцяти місцях: уся навігація оболонки, список репозиторіїв,
завантаження експорту.

Видима частина — помилка в консолі. **Невидима гірша:** якір, стилізований під
кнопку, оголошував себе посиланням. Тест це показав буквально —
`getByRole("button")` не знаходив елемент.

### Чому фікс в обгортці, а не в місцях виклику

Можна було дописати `nativeButton={false}` у дванадцяти місцях. Тоді тринадцятий
`render={<Link/>}` знову приніс би той самий баг.

Обгортка `components/ui/button.tsx` — наш файл, і вона тепер виводить
`nativeButton` з того, що передано в `render`. Явно переданий `nativeButton`
досі перемагає, а `render`-функцію (її вміст неможливо перевірити наперед)
лишено на розсуд Base UI.

Це другий свідомий локальний правок згенерованого shadcn-компонента; перший —
`checkbox.tsx`. Обидва описані в `.claude/ARCHITECTURE.md`.

### Перевірка в реальному браузері

Тестів було мало — симптом же в консолі. Тому підняв headless Chrome через CDP і
зняв консоль на `/local`:

- з фіксом → **0** попереджень
- фікс відкочено → **4** попередження
- фікс повернуто → **0**

Нуль сам по собі нічого не доводив би: спершу треба показати, що зонд узагалі
здатен це зловити. Тому середній рядок у списку — не формальність, а суть
перевірки.
